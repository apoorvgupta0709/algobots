# Trading Knowledge + Algo Development Plan

> **For Hermes:** Use this as the operating plan for turning books, market data, fundamentals, technicals, sentiment, and FYERS data into auditable trading research. Live order placement stays disabled-by-default and requires explicit user confirmation of exact order details.

**Goal:** Build a structured trading research system that stores book knowledge, market/fundamental/sentiment data, strategy hypotheses, backtests, paper trades, and approved live-trade decisions in PostgreSQL.

**Architecture:** Keep raw data immutable, extracted knowledge auditable, and algorithms versioned. Every strategy must pass through: source evidence → hypothesis → implementation → backtest → paper/live-sim → risk review → user-approved execution. No autonomous live trades without explicit confirmation.

**Tech Stack:** PostgreSQL 17, Python, FYERS API v3, local files under `/opt/data/finance-db`, future OCR/document tools for books, scheduled Hermes jobs for ingestion/reporting.

---

## 1. Safety and operating rules

1. **Read-only by default:** FYERS order book and positions may be read for analysis, but placing/modifying/cancelling orders requires explicit confirmation.
2. **Confirmation format for live orders:** symbol, side, order type, quantity, price/trigger if any, product type, validity, account, stop-loss/exit plan.
3. **No hidden discretion:** If a model recommends an action, store the reason, data snapshot, strategy version, and risk metrics.
4. **Paper-first:** Every new strategy must run in backtest and paper/simulation before live use.
5. **Capital/risk gates:** Define max capital per trade, max daily loss, max portfolio exposure, and cooldown rules before enabling any execution workflow.
6. **Not professional advice:** Outputs are research and decision support; user remains final decision-maker.

## 2. Storage design

### 2.1 Existing schemas

Already created:

- `finance.*`: personal finance accounts and transactions.
- `market.instruments`: tradable symbols.
- `market.candles`: OHLCV candles from FYERS.
- `market.quotes`: latest quote snapshots.
- `market.ingestion_runs`: audit trail for ingestion jobs.

### 2.2 New schema: `knowledge`

Purpose: store books, notes, extracted principles, setups, and evidence.

Planned tables:

- `knowledge.sources`
  - Stores book/document metadata: title, author, edition, file path, source type, uploaded_at.
- `knowledge.chunks`
  - Stores chunked text from books with page/chapter references.
- `knowledge.concepts`
  - Stores extracted concepts like trend following, position sizing, breakout failure, market breadth.
- `knowledge.rules`
  - Stores actionable rules from books, with source references and confidence.
- `knowledge.playbooks`
  - Stores trading playbooks derived from multiple rules: entry, exit, invalidation, risk, market regime.
- `knowledge.embeddings` or external vector index later
  - Optional semantic search layer for book chunks and notes.

Principle: book-derived knowledge must be traceable back to page/chapter/source.

### 2.3 New schema: `research`

Purpose: convert knowledge into testable strategy hypotheses.

Planned tables:

- `research.hypotheses`
  - Plain-English hypothesis, source rules, target universe, timeframe, expected edge.
- `research.strategy_versions`
  - Versioned strategy code/config, parameters, assumptions, status.
- `research.backtest_runs`
  - Backtest metadata: universe, dates, costs, slippage, metrics, code version.
- `research.backtest_trades`
  - Individual simulated trades.
- `research.factor_snapshots`
  - Technical/fundamental/sentiment factors used at decision time.
- `research.model_outputs`
  - Any LLM/model analysis output, stored with prompt/version/data context.

### 2.4 New schema: `trading`

Purpose: read order book/positions and store decisions/execution audit.

Planned tables:

- `trading.positions_snapshots`
  - Periodic FYERS positions snapshot.
- `trading.orderbook_snapshots`
  - Periodic FYERS order book snapshot.
- `trading.trade_ideas`
  - Candidate trades generated by strategies or analysis.
- `trading.approvals`
  - User confirmations for live trades.
- `trading.execution_log`
  - Orders placed/cancelled/modified, API status, timestamps, masked account references.

Important: order execution code should refuse to run unless there is a matching approval record.

## 3. Knowledge ingestion from books

### Step A: Store original files

- Put books/docs under:
  - `/opt/data/finance-db/library/books/`
  - `/opt/data/finance-db/library/notes/`

### Step B: Extract text

For PDFs/images:

- Use OCR/document extraction tools.
- Preserve page numbers and chapter boundaries.
- Store original file hash for audit.

### Step C: Chunk text

Chunk by:

- book → chapter → section → page range
- target chunk size: roughly 500-1,500 words
- keep overlap for context

### Step D: Extract structured knowledge

For each chunk, extract:

- concepts
- trading rules
- risk rules
- market regime assumptions
- examples
- warnings/failure modes
- quotes with page references

### Step E: Human review labels

Each extracted rule gets status:

- `draft`
- `reviewed`
- `accepted`
- `rejected`
- `converted_to_hypothesis`
- `converted_to_skill`

Only reviewed/accepted rules should influence strategy generation.

## 4. Building Hermes trading skills from books

A “skill” should be created only after repeated use and validation. Books go first into the knowledge DB, not directly into Hermes skills.

Skill candidates:

1. **technical-analysis-playbook**
   - How to analyze trend, support/resistance, volume, breakouts, failed breakouts.
2. **fundamental-analysis-playbook**
   - How to evaluate business quality, growth, leverage, margins, valuation.
3. **sentiment-analysis-playbook**
   - How to interpret news/social/market sentiment without overreacting.
4. **risk-management-playbook**
   - Position sizing, stop-loss logic, max loss, correlation, drawdown response.
5. **strategy-backtesting-playbook**
   - How to turn a trading idea into a tested strategy.
6. **trade-review-playbook**
   - How to journal and evaluate trades after entry/exit.

Rule: A Hermes skill should contain procedures, not raw book text. Raw text remains in `knowledge.chunks`.

## 5. Analysis layers

### 5.1 Technical analysis

Inputs:

- FYERS candles: daily/intraday OHLCV
- latest quotes
- market breadth if available

Computed factors:

- trend: moving averages, higher-high/higher-low structure
- momentum: RSI, MACD, ROC
- volatility: ATR, realized volatility
- volume: relative volume, OBV, delivery/volume spikes if available
- levels: swing highs/lows, support/resistance, breakout zones
- regime: trending/ranging/high-volatility/low-volatility

Storage:

- `research.factor_snapshots`
- `research.model_outputs`

### 5.2 Fundamental analysis

Inputs:

- financial statements
- ratios
- earnings data
- corporate actions
- sector/industry metadata

Computed factors:

- revenue/profit growth
- margins
- debt/equity
- cash flow quality
- ROE/ROCE
- valuation: P/E, P/B, EV/EBITDA where available
- promoter/institutional holding if available

Storage:

- future `fundamental.*` schema or `research.factor_snapshots` initially.

### 5.3 Sentiment analysis

Inputs:

- news headlines/articles
- exchange filings/announcements
- social/media feeds where available
- market reaction after news

Computed factors:

- sentiment score
- event type: earnings, management change, regulation, order win, litigation, macro
- credibility/source weight
- novelty vs repeated news
- contradiction flags

Storage:

- `sentiment.events`
- `sentiment.scores`
- `research.factor_snapshots`

## 6. Strategy development lifecycle

Each strategy follows this lifecycle:

1. **Knowledge source**
   - Rule/playbook extracted from books or research.
2. **Hypothesis**
   - Example: “Breakouts with rising relative volume and positive sector trend outperform over 10 trading days.”
3. **Specification**
   - Universe, timeframe, indicators, entry, exit, stop, position sizing, transaction costs.
4. **Backtest**
   - Store metrics and all simulated trades.
5. **Robustness checks**
   - Walk-forward, parameter sensitivity, slippage, transaction costs, out-of-sample period.
6. **Paper trade**
   - Run live signal generation without placing orders.
7. **Review**
   - Compare actual vs expected behavior.
8. **Approval for live use**
   - User approves strategy and risk limits.
9. **Live execution with confirmation**
   - Still requires explicit confirmation per order unless user later defines a bounded automation policy.
10. **Post-trade journal**
   - Store outcome, screenshots/data snapshot, rule adherence, lessons.

## 10. Non-negotiable verification

Before a strategy can be considered usable:

- every signal has source data snapshot
- every rule has source/evidence or strategy version
- every backtest stores full trade list
- every live action requires approval record
- every FYERS API call logs status and response metadata
- no strategy can place orders directly from LLM text
